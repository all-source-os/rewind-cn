Why Rewind?

CQRS Engine Core

Command-Query Responsibility Segregation separates writes (commands that emit events) from reads (projections that materialize views). Clean, testable, auditable.

Event Sourcing Events

Every state change is an immutable event. Full audit trail, time-travel debugging, and reproducible state. Powered by the AllSource embedded event database.

Clean Architecture Core

Strict dependency direction: domain (zero deps) → application (pure logic) → infrastructure (framework). Easy to test, extend, and reason about.

Agent Workers Agent

Spawn concurrent AI coding agents per task. Each agent gets a focused context window with clear acceptance criteria. First-write-wins prevents conflicts.

Chronis Integration Agent

Uses Chronis as the task tracker. Chronis owns the backlog, Rewind owns execution. Bridge via cn ready, cn claim, cn done.

MCP Server Agent

Expose capabilities as MCP tools and resources for IDE integration. Supports optional --toon format for ~50% fewer tokens in agent workflows.

Multi-Provider LLM Agent

Per-role provider configuration: use Anthropic for coding, OpenAI for planning, or any mix. Each role (planner, coder, evaluator) independently configurable. Powered by rig-core.

TUI Dashboard Core

Live terminal dashboard during execution with --tui. See task progress, agent activity, and epic completion in real time.

Git Worktree Parallel Agent

Isolate each agent in its own git worktree with --parallel. True parallel execution without merge conflicts. Changes merge back on task completion.

Beads Import Core

Import task hierarchies from chronis/beads JSONL files. Extracts acceptance criteria, quality gates, and dependency graphs automatically.

Execution Analytics Events

Query the event store for task timing, tool usage patterns, session history, and epic progress. Built on the same projections that power status.

Skill Pipeline

End-to-end automation with Claude Code skills

1

/rewind-prd — Generate PRD

Describe a feature. The skill asks clarifying questions, classifies quality gates into epic-level vs. story-level, and outputs a structured PRD with verifiable acceptance criteria.

2

/rewind-beads — Create Beads

Convert the PRD to chronis beads. Detects project tooling (Cargo, Taskfile, Makefile), creates an epic with quality gates, child beads for each story, and dependency ordering.

3

rewind run --tracker chronis — Execute

Rewind picks unblocked beads, spawns agents, tracks progress via events. Agents verify each acceptance criterion, mark items done, and close beads. Epic-level gates run on completion.

Commands

Command Description
rewind init Create .rewind/ directory with config and event store
rewind plan <desc> Generate an execution plan (epic + tasks) from a description
rewind plan -f <file> Generate a plan from a PRD markdown file
rewind run Execute pending tasks through agent workers
rewind run --parallel --tui Parallel worktrees + live TUI dashboard
rewind status Display backlog and epic progress
rewind query <name> Query analytics (task-summary, epic-summary, tool-usage, session-history)
rewind import <file> Import epics/tasks from beads JSONL/JSON files
rewind report Export anonymized diagnostic report
rewind mcp Start MCP server over stdio for IDE integration

Claude Code Skills

Install the bundled skills for end-to-end automation

/rewind-prd

Generate structured PRDs optimized for AI agent execution. Asks clarifying questions, classifies quality gates, outputs verifiable acceptance criteria.

  • Iterative clarifying questions with lettered options
  • Two-tier quality gates (epic-level vs. story-level)
  • Type-tagged user stories: [Domain], [Application], [Infrastructure], [CLI]
  • Verifiable - [ ] acceptance criteria
  • Wrapped in [PRD]...[/PRD] markers

/rewind-beads

Convert PRDs to chronis beads for autonomous execution. Detects project tooling and creates dependency-ordered tasks.

  • Auto-detects Cargo, Taskfile, Makefile, package.json
  • Creates epic bead with epic-level quality gates
  • Child beads per user story with cn create --toon
  • Dependency ordering: domain → application → infrastructure → CLI
  • Right-sized stories (one agent context window)
